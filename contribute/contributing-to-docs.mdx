# Contributing to the docs repo

This page covers how contribution mechanics work specifically in the `docs` repo - the PR flow, what CODEOWNERS review means here, and where the branch protection rules live. For the general edit-and-submit workflow, see the [contribution guide][1].

## Open a PR against `docs`

Push your branch and open a pull request as usual. A few rules on `main` apply that aren't the GitHub default:

* **Squash or rebase merges only.** `main` requires a linear history, so merge commits aren't allowed - use **Squash and merge** or **Rebase and merge** on the PR.
* **All review conversations must be resolved** before the PR can merge.
* **At least one approving review from a Code Owner is required.**
* Force-pushes to `main` and branch deletion are blocked.

<Note>
GitHub blocks self-approval of your own PR at the API level. Combined with CODEOWNERS below, this means the PR author can never satisfy their own review requirement.
</Note>

## What CODEOWNERS review means in practice

[`.github/CODEOWNERS`][2] currently reads:

```text
* @digitaldiina
```

One person owns approval rights for the entire repo. In practice:

* Anyone else's PR needs `@digitaldiina`'s approval before it can merge.
* `@digitaldiina`'s own PRs can't get a normal Code Owner approval - self-approval is blocked, and there's no second Code Owner yet to approve instead.

That's why `main`'s branch protection currently has **admin enforcement turned off**: it lets the one non-owner repo admin (`@digitaldiina`) bypass the review requirement and merge directly, without opening the review gate to anyone else. This is intentional for the pre-launch period, not a permanent setup - expect it to tighten up (more Code Owners, admin enforcement back on) after go-live.

## Where to find the branch protection rules

The live rule is on GitHub under **Settings → Branches** for the `main` branch. As of this writing, it requires: 1 Code Owner approval, resolved conversations, and linear history, with force-pushes and branch deletion blocked and admin enforcement off (see above).

<Note>
Org owners of `SuperOfficeDocs` bypass all branch protection on every repo in the org unconditionally - this isn't a per-repo setting and can't be changed here.
</Note>
